一、Git 简介

1.1 Git 是什么？

  • Git 是目前世界上最先进的分布式版本控制系统。

    • Workspace：工作区，就是电脑中项目所在目录。

    • Index / Stage：暂存区，用于存放临时改动，一般存放在 .git 目录下的 index 文件中，所以我们把暂存区有时也叫作索引（index）。

    • Repository：仓库区（版本库），

    • Remote：远程仓库。

1.2 Git 工作流程

  • git 的工作流程：

    • 刚开始 Workspace、 Index 与 Repository(HEAD) 三者的內容是一致的。

    • 当 git 管理的项目内容发生变化时，Workspace 的內容就会与 Index 和 Repository(HEAD) 不一致，此时 git 就会将被修改的文件（Tracked File）状态设置为 modified（Unstaged files）。

    • 执行 git add 后，会将这些改变的文件內容加入到 Index 中，文件状态变为 staged（Staged files），此时 Workspace 与 Index 的內容一致，但他们与 Repository(HEAD) 的內容不一致。

    • 执行 git commit 后，Index 中所有改变的文件內容被提交至 Repository(HEAD)，建立新的 commit 节点（HEAD），文件状态变为 committed（Committed files），此时 Workspace、 Index 与 Repository(HEAD) 三者的內容重新保持一致。

    • 执行 git push 后，Remote 远程仓库中的内容就会与 Workspace 同步。

1.3 Git 文件状态

  • Git 管理的文件有三种状态：已修改（modified），已暂存（staged），已提交（committed）。

二、Git 与 SVN 区别

  • SVN 是 集中式版本控制系统，版本库是集中放在中央服务器的，而干活的时候，用的都是自己的电脑，所以首先要从中央服务器哪里得到最新的版本，然后干活，干完后，需要把自己做完的活推送到中央服务器。而且集中式版本控制系统是必须联网才能工作。

  • Git 是 分布式版本控制系统，它是没有中央服务器的，每个人的电脑就是一个完整的版本库，这样，工作的时候就不需要联网了，因为版本都是在自己的电脑上。既然每个人的电脑都有一个完整的版本库，那多个人如何协作呢？例：自己在电脑上改了文件 A，其他人也在电脑上改了文件 A，这时，你们之间只需把各自的修改推送给对方，就可以互相看到对方的修改了。

五、Git 操作

5.1 创建版本库

  • 版本库（repository）又名仓库，即本地仓库，可以看做一个目录，这个目录里的所有文件都由 Git 进行管理，目录中每个文件的修改、删除，Git 都能跟踪，以便任何时刻都可以追踪历史，或者在将来某个时刻可以 “还原”。

  • 创建版本库的步骤：

    • 创建空目录 mkdir repository

    • 通过 git init 命令来初始化一个版本库，在执行 git init 命令后，Git 仓库会生成一个 .git 目录（默认隐藏），这个目录是 Git 用来跟踪管理版本的。有两种初始化的方式。

      • 1）在指定目录中使用 git init 初始化

      • 2）指定要初始化的目录。例：git init E:\repository

5.2 添加 / 删除文件（工作区 <–> 暂存区）

5.2.1 git add 添加文件

  • git add .：提交新文件（new）和被修改（modified）的文件，不包括被删除（deleted）的文件。

  • git add dir：添加指定目录 dir 中所有的文件到暂存区。

  • git add *.txt：使用通配符 *，添加所有后缀为 txt 的文件。

  • git add file1 file2 ...：添加指定的一个或多个文件到暂存区。

  • git add -A：提交所有变化。（-A == --all）

  • git add -u：提交被修改（modified）和被删除（deleted）的文件，不包括新文件（new）。（-u == --update）

  • git add -h：查看 git add 相关帮助。（h == --help）

5.2.2 git rm 删除文件

  • git rm file：同时将文件从工作区和版本库删除，删除操作被放入暂存区。

    • 注：被删除的文件必须是没有修改过的，即与当前版本库文件的内容相同。

  • git rm -f file：同时将文件从工作区和暂存区删除，删除操作被放入暂存区。

    • 注：文件在删除之前被修改过，即与当前版本库文件的内容不同，必须使用强制删除选项 -f。

  • git rm --cached file：将文件从版本库删除，删除操作被放入暂存区，本地文件依然存在，只是不希望这个文件被版本控制。

5.2.3 rm 删除文件

  • rm file 只会删除工作区的文件。如果想在版本库中同步删除，需要 git add file，然后 git commit file。

5.3 提交文件（暂存区 <–> 版本库）

  • 使用 git commit 命令将添加到暂存区的文件提交到版本库。

    • git commit -m [注释可选]：提交暂存区的所有文件和操作到版本库。

    • git commit [file1] [file2] -m [注释可选]：提交暂存区的指定文件和操作到版本库。

    • git commit -a -m [注释可选]：提交工作区自上次 commit 之后的所有变化，直接到版本库，不需要执行 git add 命令。

      • -a 参数相当于执行了 git add -u 命令。

    • git commit -v -m [注释可选]：提交时显示所有 diff 信息。

    • git commit --amend -m [注释可选]：使用一次新的 commit，替代上一次提交，如果代码没有任何新变化，则用来改写上一次 commit 的提交信息，即注释。

5.4 回退版本

  • git reset 命令用于回退版本，可以指定退回某一次提交的版本。

  • git reset 语法：git reset [--soft | --mixed | --hard] [HEAD] [filename]

5.4.1 –soft 参数

  • --soft 参数用于回退到某个指定的 [HEAD] 版本，此模式下会保留 Workspace 工作区的內容，并将因为保留工作区所带来的新的文件差异放进 Stage 暂存区。

    • 该模式撤销 commit，不撤销 git add .，恢复到指定版本的 add 状态。

    • 该模式不会改变 git reset 前所有的 git 管理的文件夹的內容，即保留 Workspace 工作区的內容，同时也会保留 Stage 暂存区的內容，让 Stage 暂存区与 Workspace 工作区的內容保持一致的。

5.4.2 –hard 参数

  • --hard 参数用于回退到某个指定的 [HEAD] 版本，此模式下会将 Workspace 工作区、Stage 暂存区和 Repository 版本库都回退到某个指定版本。

    • 该模式撤销 commit，撤销 git add .，恢复到指定版本的 commit 状态。

    • --hard 和 --soft 的区别：--hard 会清空工作区和暂存区的改动，而 --soft 则会保留工作区的内容，并把因为保留工作区内容所带来的新的文件差异放进暂存区。

5.4.3 –mixed 参数

  • --mixed 为默认参数，用于回退到某个指定的 [HEAD] 版本，此模式下会保留 Workspace 工作区的內容，并且清空 Stage 暂存区。

    • 撤销 commit，并且撤销 git add .，恢复到指定版本的 commit 状态。。

    • --mixed 和 --soft的区别：--hard 会清空暂存区，而 --soft 会把因为保留工作区内容所带来的新的文件差异放进暂存区。

5.4.4 HEAD 参数

  • HEAD 参数用于指定回退版本，有两种表示方法：

    • 上箭头 ^ 方法：

      • HEAD 表示当前版本。

      • HEAD^ 表示前 1 个版本。

      • HEAD^^ 表示前 2 个版本。

      • 以此类推 $\cdots$

    • ～num 方法：

      • HEAD~0 表示当前版本。

      • HEAD~1 表示前 1 个版本。

      • HEAD^2 表示前 2 个版本。

      • 以此类推 $\cdots$

5.5 查看状态

  • git status 命令用于查看在你上次提交之后是否有对文件进行再次修改。

  • git ls-files 命令用于输出暂存区的文件信息，或输出暂存区与工作区之间的差异化信息。

5.6 分支操作

  • git branch：查看本地所有分支，当前所在分支前会有一个星号 *。

  • git branch -r：查看远程所有分支。

  • git branch -a：查看本地和远程所有分支。

  • git checkout branchname：切换到 branchname 分支。

  • git branch [-f] branchname：新建一个分支，但不切换。

    • 可以使用 git checkout -b branchname 新建并切换至新分支。

  • git branch -M custom_branch：修改当前项目的分支为 custom_branch 分支。例：

    • git branch -M main 修改当前项目的分支为 main 分支。

  • git config --global init.defaultBranch branchname：修改本地仓库默认分支为 branchname 分支。例：

    • git config --global init.defaultBranch main 修改默认分支为 main 分支。

  • git branch (-d | -D) branchname：删除本地分支。

  • git branch (-d | -D) -r branchname：删除远程分支。

  • git branch (-m | -M) oldbranch newbranch：重命名分支。

  • git merge branchname：在当前分支合并 branchname 分支。

六. 远程操作

  • 我们已经在本地创建了一个 Git 仓库（Repository），现在想要在 Github 上创建一个远程仓库进行代码托管，并且希望这两个仓库实现远程同步，这时就需要进行远程操作。

6.1 Github 创建远程仓库

  • 登录 Github，然后在右上角找到 “New repository”，创建一个新的仓库。

  • Repository name 填写要创建的远程仓库名称，其他默认，然后点击 Create repository 创建仓库。

6.2 本地仓库 与 Github 远程仓库关联

  • 在本地仓库中运行指令：

    • 运行 git remote add <shortname> <url> 添加一个新的远程 Git 仓库

      • 远程仓库命名为 origin 是 Git 的默认叫法，可以更改。

  • git remote -v 命令用于显示所有远程仓库

  • git remote show 仓库地址 命令用于显示远程仓库的信息。

6.3 文件上传版本库

  • 将需要进行版本管理的文件通过 git add 放入 Stage 暂存区。

  • 将暂存区的文件通过 git commit 提交到 Repository 版本库。

6.4 文件推送

  • git push <远程仓库名> <本地分支名>:<远程分支名> 命令用于将本地仓库推送到远程仓库。

    • 如果本地分支名与远程分支名相同，则可以写作 git push <远程仓库名> <本地分支名>。

    • 当我们第一次推送时加上 -u 参数（git push -u），Git 不但会把本地分支内容推送到远程分支，还会把本地分支和远程分支关联起来，在以后的推送或者拉取时可以简化命令，不加任何参数。

  • 建议在 push 前先使用 git pull <远程仓库名> <远程分支名> 拉取远程仓库，保持本地仓库与远程仓库的内容一致。

    • 如果报错 fatal: refusing to merge unrelated histories，可以在 pull 命令后添加 --allow-unrelated-histories 参数来解决问题，该参数可以合并两个独立仓库的历史。

  • git push --force <远程仓库名> <本地分支名>：当本地版本与远程版本有差异，但又要强制推送可以使用 --force 参数。

  • git push <远程仓库名> --delete <远程分支名>：当删除远程仓库的分支时可以使用 --delete 参数。

6.5 SSH 警告

  • 第一次使用 Git 的 clone / push / 命令连接 GitHub 时，会得到一个警告：

    • 这是因为 Git 使用 SSH 进行远程连接，而 SSH 连接在第一次验证 GitHub 服务器的 Key 时，需要确认 GitHub 的 Key 是否来自 GitHub服务器，输入 yes 回车即可。

    • 输入 yes 回车后，Git 会输出一个警告，告诉你已经把 GitHub 的 Key添加到本机的一个信任列表。

      • 这个警告只会出现一次，后面的操作就不会有任何警告了。

6.6 从远程仓库克隆

  • 当我们想要将远程仓库中已经存在的仓库下载到本地时，可以使用命令 git clone 进行克隆。

  • 格式：git clone <远程仓库地址> <本地目录名>

    • 远程仓库地址：git clone 支持多种协议，除了 HTTP(s) 以外，还支持 SSH、Git、本地文件协议等。

    • 本地目录名：远程仓库下载到本地版本库的名称，默认与远程仓库同名。
